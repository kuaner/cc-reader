# Design Context — CC Reader

## Design Context

### Users

在 macOS 上打开 **Claude Code** 会话 JSONL 的开发者与高级用户。核心场景是：**快速浏览整条 session、对照工具调用 / thinking / dispatch、理解 agent 实际做了什么**，并在出问题时**定位片段、排查原因**——不是以「沉浸式长文阅读」为第一目标。

### Brand Personality

**冷静、可信赖、偏工具感**：像**检视器（inspector）**或**运行记录面板**，而不是聊天玩具或消费型阅读器。界面应支持**扫视、跳转、对照**，而不是鼓励慢读散文。

### Aesthetic Direction

- **Session Ledger（会话档案）**：视觉上仍可用暖中性 + 用户侧强调色区分角色，但语义上是**可追溯的执行记录**，不是书页。浅色下助手为淡暖灰纸、用户为锈橙系；深色下助手偏冷板岩、用户偏暖褐；用户侧保留细左边线与 rim，便于在密集列表里一眼扫到「人的输入」。
- **避免**：蓝青霓虹、紫蓝渐变、暗色发光描边等典型「AI 产品」套路（见 frontend-design skill）。
- **主题**：浅色与深色均支持，随系统 `prefers-color-scheme`。
- **参考气质**：调试器、日志查看器、技术向 Session 时间轴——**清晰、紧凑、可扫**；不是社交 IM，也不是文学阅读器。

### Design Principles

1. **检视优先于阅读**：默认偏**紧凑**、信息密度高；字号与间距首先服务**扫错与对照**，长文可读性在其次（仍不牺牲基本对比度与 WCAG 可接受范围）。
2. **结构先于装饰**：助手报头、工具块、thinking、标签等层级要一眼可辨；装饰性阴影/圆角克制。
3. **单一强调色**：用户侧与焦点环共享暖色强调，便于在时间线里快速对齐「人类输入」与交互目标。
4. **Token 驱动**：颜色只进 `timeline-ui/src/styles/design-tokens.css` 语义变量（时间线与 markdown 预览共用），组件不写死 hex。
5. **可访问**：链接与焦点可见；动效尊重 `prefers-reduced-motion`（沿用现有）。
6. **字体分工**：时间线 Web 栈使用 **Apple 系统字体**（`-apple-system` / SF，见 `timeline-ui` `--font-ui`）；Markdown 与 UI 同栈，代码块用等宽栈；不依赖网络字体加载。
7. **对齐与密度**：报头标签与模型 chip 等控件**同基准线/同高度档**，避免「大点击区组件」与检视行混用导致错位；列表与气泡间距偏紧，利于一屏多看几条。

## Onboarding & First-Time Experience（检视器导向）

以下指导 **空状态、首次进入时间线、产品文案与渐进披露**，与 frontend-design / onboard 技能一致；**本产品不是「教会用户如何阅读」**，而是**尽快让人会「查 session、跟 agent、排错」**。

### 核心目标（Aha moment）

用户应在短时间内理解：**这条时间线在展示什么、从哪里看出工具调用 / 思考 / 用户输入、如何复制或对照原数据**。  
成功标准：**能独立在一条 session 里定位「哪一步 tool、哪一段 thinking、哪条用户指令」**，而不是「读得舒服」。

### 用户与耐心

- **偏熟练用户**：多数会跳过冗长引导；**不要**在能用产品前强制多步教程。
- **时间预算**：首次有价值交互宜在 **1–2 分钟内**（打开会话 → 扫到结构 → 展开/复制一条）。
- **替代方案**：用户可能来自终端、IDE、其它 log 工具——文案避免假设「你是来读故事的」。

### 空状态与时间线无消息

| 场景 | 应传达的信息 | 语气 |
|------|----------------|------|
| **尚未加载会话 / 无消息** | 这里将显示该 session 的消息与元数据；可提示「选择会话」或「等待同步」（由宿主决定具体 CTA） | 短句、工具向，**不要**用「开始阅读」「享受对话」 |
| **搜索/筛选无结果** | 无匹配项；建议改条件或清空筛选 | 陈述事实 + 可操作下一步 |
| **加载失败** | 失败原因 + 重试；不 blame 用户 | 与调试器一致：可恢复、可重试 |

**实现注意**：时间线内容由原生侧注入 `timeline-shell.html` 占位；若需空状态 HTML，应由 **Swift 在 `messageHTML` 为空时注入**一段简短说明（便于 **Localizable**），避免在 CSS 里写死长文案。

### 文案与微文案原则

- **说「检视 / 记录 / 运行 / 工具 / 消息」**，少说「阅读 / 故事 / 沉浸」。
- **按钮与标签**：复制、原数据、加载更早、模型名——偏**操作与事实**，不煽情。
- **尊重智力**：不解释显而易见的 IM 气泡隐喻；只解释 **cc-reader 特有**行为（如 JSONL、会话文件路径等放在文档或 `?`，不在时间线里长篇大论）。

### 渐进披露（别一次教完）

- **首屏**：时间线结构（用户 vs 助手、标签、页脚操作）应自解释；其余交给 **快捷键表（README / 帮助）** 与 **上下文菜单**。
- **新功能**：用「首次出现时的轻提示」优于全屏导览；可 **Don't show again**。
- **进阶**：Context 面板、会话操作等——在用户打开对应区域时再提示一句即可。

### 明确不做（与 onboard 技能对齐）

- 不阻断主流程的长篇 **Welcome 阅读器式** 介绍。
- 不把 **紫蓝渐变 / 玻璃拟态** 当「欢迎页」视觉（见 frontend-design）。
- 不重复展示用户已点过「不再显示」的提示（若以后加 LocalStorage / 偏好，需尊重）。

### 验收（可抽样自测）

- 新用户能否在 **不读长文** 的情况下说出：**哪条是用户、哪条是助手、工具结果在哪**？
- 空状态是否说明 **「这里会出现什么」** 而非 **「欢迎使用阅读体验」**？
- 熟练用户是否能在 **10 秒内** 跳过或忽略任何引导？
